## 一、 引言：为何需要多文件编译？

在初学阶段，我们习惯将所有代码写在单文件（如 `main.cpp`）中。然而，随着项目复杂度的提升（例如引入图像预处理、特征检测、机器学习模型等多个模块），单文件模式会导致代码逻辑混乱、难以维护。

**多文件编译的核心优势包括：**
1. **模块化 (Modularity)**：将功能逻辑解耦，不同开发人员可以并行负责不同的模块。
2. **增量编译 (Incremental Build)**：修改某一模块时，只需重新编译该模块对应的源文件，大大缩短大型项目的构建时间。
3. **结构清晰**：符合“高内聚、低耦合”的软件工程思想。

## 二、 C++ 多文件协作的基础：头文件与源文件

在 C++ 工程实践中，多文件协作通常遵循以下标准范式：

* **头文件 (`.h` / `.hpp`)**：负责**声明**。包含函数原型、类结构定义、宏定义等。
    * *关键点*：必须使用 `#ifndef` / `#define` / `#endif`（Include Guards）或 `#pragma once` 防止重复包含。
* **源文件 (`.cpp`)**：负责**实现**。编写具体的函数体和逻辑代码。
* **主入口 (`main.cpp`)**：通过 `#include` 引入头文件，调用各个模块的功能。

## 三、 多文件编译的三种主流实现方案

### 方案 1：基于 g++ 命令行手动编译（底层机制）
这是最本质的编译方法，适合理解编译器与链接器的工作原理。

* **操作命令**：
    ```bash
    g++ main.cpp module1.cpp module2.cpp -o my_vision_app \
        -I /usr/include/opencv4 \
        -l opencv_core -l opencv_imgcodecs -l opencv_highgui -l opencv_imgproc
    ```
* **技术原理**：
    编译器首先将每个 `.cpp` 文件分别翻译成二进制的目标文件（`.o` 或 `.obj`），随后**链接器 (Linker)** 负责寻找函数调用与定义之间的地址映射，最终打包成可执行文件。

### 方案 2：基于 VS Code `tasks.json` 的自动化配置（开发效率方案）
在 IDE 环境中，我们可以通过修改配置文件，让 VS Code 自动寻找并编译目录下的所有源文件。

* **配置核心改动**：
    在 `.vscode/tasks.json` 的 `args` 数组中，将特定的文件名替换为**通配符**：
    ```json
    "args": [
        "-g",
        "${workspaceFolder}/src/*.cpp", // 核心：自动包含 src 目录下所有源文件
        "-o", "${workspaceFolder}/build/out",
        // 链接 OpenCV 库...
        "-l", "opencv_core", "-l", "opencv_imgcodecs"
    ]
    ```
* **适用场景**：适合中小型课程作业，无需安装额外的构建工具，一键 F5 即可运行。

### 方案 3：基于 CMake 的项目构建系统（工业界标准）
CMake 是目前 C++ 领域最主流的跨平台构建系统，能够自动处理复杂的库依赖和文件关联。

* **`CMakeLists.txt` 编写示例**：
    ```cmake
    cmake_minimum_required(VERSION 3.10)
    project(OpenCV_App)

    # 1. 自动寻找 OpenCV 库环境
    find_package(OpenCV REQUIRED)

    # 2. 搜索指定目录下的所有源文件并存入变量
    aux_source_directory(./src DIR_SRCS)

    # 3. 指定编译目标
    add_executable(my_app ${DIR_SRCS})

    # 4. 自动链接 OpenCV 的所有依赖库
    target_link_libraries(my_app ${OpenCV_LIBS})
    ```
* **优势分析**：极强的可移植性（Windows/Linux 切换无压力），支持大型工程的自动化管理，是处理 OpenCV 复杂依赖的最佳实践。

## 四、 实验总结与反思

通过本次作业的实践，我深刻体会到：
1.  **链接错误 (`undefined reference`)** 通常是由于源文件未包含在编译命令中，或者静态/动态库链接路径 (`-l` 参数) 缺失导致的。
2.  **构建工具的进化**：从手动输入命令到 `tasks.json` 脚本，再到 `CMake` 系统，反映了软件开发从“关注细节”向“关注架构”的演进。
3.  **实践建议**：对于简单的 OpenCV 任务，使用 VS Code 的通配符编译已足够；但对于未来涉及深度学习框架或大型 CV 算法库的项目，必须熟练掌握 CMake 的使用。
